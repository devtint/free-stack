01

Bot Setup & BotFather Config

Every Telegram bot begins with a conversation with @BotFather — Telegram's official bot management bot. This gives you an API token which is the master key to your bot.

Step-by-step: Creating your bot

1

Open @BotFather

In Telegram, search for @BotFather and start a chat. Send /start to see all commands.

2

Create a new bot

Send /newbot. BotFather will ask for a display name (any name), then a username that must end in bot (e.g. myapp_bot).

3

Save your token

You receive a token like 123456:ABCdef.... Treat this like a password — it grants full control of your bot.

4

Configure bot settings

Use /setdescription, /setabouttext, /setuserpic, and /setcommands to polish your bot's profile.

5

Disable groups (optional)

For private bots, send /setjoingroups → Disable to prevent your bot from being added to groups.

Minimal Python bot skeleton

Using the python-telegram-bot library (v20+, async). Install: pip install python-telegram-bot

bot.py
import os from telegram import Update from telegram.ext import Application, CommandHandler, MessageHandler, filters BOT_TOKEN = os.getenv("BOT_TOKEN") # Never hardcode! async def start(update: Update, context) -> None: await update.message.reply_text("👋 Hello! I'm alive.") async def echo(update: Update, context) -> None: await update.message.reply_text(update.message.text) def main(): app = Application.builder().token(BOT_TOKEN).build() app.add_handler(CommandHandler("start", start)) app.add_handler(MessageHandler(filters.TEXT & ~filters.COMMAND, echo)) app.run_polling() # switch to run_webhook() for production if __name__ == "__main__": main()

BotFather command cheatsheet

🤖

/setcommands

Register slash commands so they appear in the menu. Format: command - description one per line.

🔒

/setprivacy

In groups: ENABLED = bot only reads messages starting with /. Recommended for most bots.

🔑

/revoke

Instantly invalidates the current token and issues a new one. Use if your token is ever leaked.

📋

/mybots

Lists all your bots with quick links to edit each one's settings, token, and inline mode config.

02

Hosting on Render

Render offers a free tier for web services and background workers — perfect for Telegram bots. The catch: free services spin down after 15 minutes of inactivity. We'll fix that.

Deployment setup

1 · Push to GitHub
Put your bot code in a GitHub repo. Render deploys directly from it on every push to main.
2 · Create a Web Service on Render
Go to render.com → New → Web Service. Select your repo. Choose Python runtime. Set Start Command.
3 · Set environment variables
Add BOT_TOKEN and any other secrets in Render's Environment panel — never commit them to git.
4 · Enable auto-deploy
Render rebuilds and redeploys your bot automatically on every push. Zero manual deploys needed.
5 · Note your service URL
Render gives you a URL like https://my-bot.onrender.com. You'll need this for the webhook.

Required files

requirements.txt
python-telegram-bot==20.7 fastapi==0.109.0 uvicorn[standard]==0.27.0 supabase==2.3.0 httpx==0.26.0
render.yaml (optional — Infrastructure as Code)
services: - type: web name: my-telegram-bot runtime: python buildCommand: "pip install -r requirements.txt" startCommand: "uvicorn main:app --host 0.0.0.0 --port $PORT" envVars: - key: BOT_TOKEN sync: false # Mark secret, fill in Render dashboard - key: SUPABASE_URL sync: false - key: SUPABASE_KEY sync: false

☠️ The spin-down problem & keep-alive trick

Free Render services sleep after 15 min of no HTTP traffic. When a Telegram message arrives, the service cold-starts (~30s delay) — the message is lost or very slow. Solution: self-ping every 10 minutes.

⚠️

Note: Render's free tier now limits free hours per month. For production bots with consistent traffic, upgrade to the $7/mo plan or use Railway / Fly.io alternatives.

main.py — FastAPI + keep-alive ping
import asyncio, os, logging import httpx from fastapi import FastAPI, Request from telegram import Update, Bot from telegram.ext import Application, CommandHandler from contextlib import asynccontextmanager TOKEN = os.getenv("BOT_TOKEN") BASE_URL = os.getenv("RENDER_EXTERNAL_URL") # auto-set by Render async def keep_alive(): """Ping ourselves every 10 min to prevent spin-down.""" async with httpx.AsyncClient() as client: while True: await asyncio.sleep(600) # 10 minutes try: await client.get(f"{BASE_URL}/health") logging.info("keep-alive ping sent") except Exception as e: logging.warning(f"keep-alive failed: {e}") ptb_app = Application.builder().token(TOKEN).updater(None).build() @asynccontextmanager async def lifespan(app: FastAPI): await ptb_app.initialize() await ptb_app.start() asyncio.create_task(keep_alive()) # start ping loop yield await ptb_app.stop() app = FastAPI(lifespan=lifespan) @app.get("/health") async def health(): return {"status": "ok"} @app.post(f"/webhook/{TOKEN}") async def webhook(request: Request): data = await request.json() update = Update.de_json(data, ptb_app.bot) await ptb_app.process_update(update) return {"ok": True}
💡

Pro tip: Use an external uptime monitor like UptimeRobot (free) to ping /health every 5 minutes instead of self-pinging. More reliable and saves you the keep-alive code entirely.

03

Webhook vs Polling

There are two ways for your bot to receive updates from Telegram. The right choice depends on your environment.

Feature 🔗 Webhook 🔄 Long Polling
How it works Telegram POSTs updates to your HTTPS URL Your bot repeatedly asks Telegram for updates
Latency Near real-time ~1–2s delay
Requires public HTTPS Yes (SSL cert needed) No
Best for Production, hosted servers Local development, testing
Server resource use Low (event-driven) Higher (continuous loop)
Scales to multiple instances Yes No (race condition)

Setting up the webhook

After deploying to Render, register the webhook with Telegram. Run this once (or on each deploy):

set_webhook.py — run once after deploy
import asyncio, os from telegram import Bot async def main(): token = os.getenv("BOT_TOKEN") base_url = os.getenv("RENDER_EXTERNAL_URL") bot = Bot(token=token) webhook_url = f"{base_url}/webhook/{token}" await bot.set_webhook( url=webhook_url, allowed_updates=["message", "callback_query"], drop_pending_updates=True, # ignore messages sent while offline ) info = await bot.get_webhook_info() print(f"✅ Webhook set: {info.url}") print(f" Pending updates: {info.pending_update_count}") asyncio.run(main())

Auto-register webhook on startup

Better approach — set the webhook automatically when your FastAPI app starts:

In your lifespan() function
@asynccontextmanager async def lifespan(app: FastAPI): await ptb_app.initialize() await ptb_app.start() # Auto-register webhook on boot await ptb_app.bot.set_webhook( url=f"{BASE_URL}/webhook/{TOKEN}", drop_pending_updates=True ) asyncio.create_task(keep_alive()) yield # Clean up on shutdown await ptb_app.bot.delete_webhook() await ptb_app.stop()
ℹ️

Local dev tip: Use ngrok http 8000 to get a public HTTPS tunnel to your local machine, then set the webhook to the ngrok URL. Perfect for testing webhooks without deploying.

04

Securing Your Bot

Security is non-negotiable in production. A vulnerable bot can be abused, spammed, or used to leak user data. Cover all three layers: token security, request validation, and rate limiting.

🔐 Token & secret management

🚨

NEVER commit your BOT_TOKEN to git. GitHub's secret scanning can catch it, but Telegram tokens are immediately exploitable. Use environment variables exclusively.

.env (local only — add to .gitignore!)
BOT_TOKEN=123456:ABCdef-your-actual-token SUPABASE_URL=https://xxxx.supabase.co SUPABASE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... ALLOWED_USERS=123456789,987654321 # optional: whitelist user IDs

✅ Validating that requests come from Telegram

Anyone who knows your webhook URL could send fake updates. Embed the token in the URL path (Telegram's recommended approach) — or validate the secret token header:

Validate with secret_token header (Telegram Bot API 6.0+)
import secrets, hashlib, hmac SECRET_TOKEN = secrets.token_hex(32) # store in env, pass to set_webhook() # In set_webhook(): await bot.set_webhook( url=webhook_url, secret_token=SECRET_TOKEN # Telegram sends X-Telegram-Bot-Api-Secret-Token header ) # In your FastAPI route: @app.post("/webhook") async def webhook(request: Request): incoming = request.headers.get("X-Telegram-Bot-Api-Secret-Token", "") if not hmac.compare_digest(incoming, SECRET_TOKEN): raise HTTPException(status_code=403, detail="Forbidden") # ... process update

⏱ Rate limiting per user

In-memory rate limiter (use Redis for multi-instance)
from collections import defaultdict import time RATE_LIMIT = 5 # max requests WINDOW = 60 # per 60 seconds _user_hits: dict = defaultdict(list) def is_rate_limited(user_id: int) -> bool: now = time.time() hits = [t for t in _user_hits[user_id] if now - t < WINDOW] _user_hits[user_id] = hits if len(hits) >= RATE_LIMIT: return True _user_hits[user_id].append(now) return False # Use in handler: async def handle_message(update: Update, context): if is_rate_limited(update.effective_user.id): await update.message.reply_text("⚠️ Slow down! Try again in a minute.") return

🛡 Additional security checklist

🚫

User allowlist

For private bots, check update.effective_user.id against an allowlist before processing any command.

🧹

Sanitize inputs

Never trust user input. Sanitize before inserting into queries. Use parameterized queries — never string concat.

📝

Log, don't print

Use Python's logging module. Never log sensitive data (tokens, full messages). Use structured logs.

🔄

Rotate tokens

If a token leaks: /revoke in BotFather immediately. Update your env var in Render. Redeploy.

05

Supabase as the Database

Supabase gives you a hosted PostgreSQL database with a generous free tier, a Python client, and row-level security. It's an ideal backend for Telegram bots that need persistent storage.

Setup

1

Create a project at supabase.com

Free tier includes 500MB DB, 1GB file storage, 50MB bandwidth/day. More than enough to start.

2

Grab your credentials

Go to Project Settings → API. Copy the URL and anon/public key. Use the service_role key server-side only.

3

Create your tables in SQL Editor

Use Supabase's SQL Editor or the Table Editor GUI to create your schema. Enable RLS on every table.

Schema: user tracking example

SQL — run in Supabase SQL Editor
-- Users table CREATE TABLE users ( id BIGINT PRIMARY KEY, -- Telegram user ID username TEXT, first_name TEXT, joined_at TIMESTAMPTZ DEFAULT NOW(), is_banned BOOLEAN DEFAULT FALSE ); -- Messages log CREATE TABLE messages ( id UUID PRIMARY KEY DEFAULT gen_random_uuid()(), user_id BIGINT REFERENCES users(id), content TEXT NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Index for fast user lookups CREATE INDEX idx_messages_user_id ON messages(user_id); -- Enable Row Level Security ALTER TABLE users ENABLE ROW LEVEL SECURITY; ALTER TABLE messages ENABLE ROW LEVEL SECURITY;

Python client integration

db.py — Supabase helper module
import os from supabase import create_client, Client from telegram import User supabase: Client = create_client( os.getenv("SUPABASE_URL"), os.getenv("SUPABASE_KEY") # use service_role key server-side ) async def upsert_user(user: User) -> None: """Create or update user record on every interaction.""" supabase.table("users").upsert({ "id": user.id, "username": user.username, "first_name": user.first_name, }, on_conflict="id").execute() async def log_message(user_id: int, content: str) -> None: supabase.table("messages").insert({ "user_id": user_id, "content": content[:4000], # trim if needed }).execute() async def is_banned(user_id: int) -> bool: result = ( supabase.table("users") .select("is_banned") .eq("id", user_id) .single() .execute() ) return result.data.get("is_banned", False) if result.data else False

Wiring it into your bot handlers

handlers.py
from telegram import Update from db import upsert_user, log_message, is_banned async def handle_message(update: Update, context): user = update.effective_user # 1. Check ban status if await is_banned(user.id): return # silently ignore banned users # 2. Upsert user profile await upsert_user(user) # 3. Log the message await log_message(user.id, update.message.text) # 4. Your bot logic here await update.message.reply_text(f"Got it, {user.first_name}!")

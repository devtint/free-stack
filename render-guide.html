01

Creating a Web Service

Render deploys any Python, Node.js, Go, Ruby, or Docker app directly from GitHub. The free tier gives you 750 hours/month — enough for one always-on web service.

Step-by-step setup

1

Push your code to GitHub

Your app must be in a GitHub repo. Render connects to GitHub and deploys on every push to your branch.

2

Go to render.com → New → Web Service

Sign in with GitHub. Click New + → Web Service. Select your repository from the list.

3

Configure runtime and commands

Render auto-detects most frameworks. Set Build Command and Start Command manually if needed (see table below).

4

Choose Free plan

Select Free instance type. Note: free services spin down after 15 min of inactivity. Fix this in Section 03.

5

Add environment variables

Go to Environment tab after creation and add your secrets. Never commit .env to git.

6

Deploy

Click Create Web Service. First deploy takes 2–5 min. You'll get a URL like https://your-app.onrender.com.

Runtime commands reference

RuntimeBuild CommandStart Command
Python (FastAPI)pip install -r requirements.txtuvicorn main:app --host 0.0.0.0 --port $PORT
Python (Flask)pip install -r requirements.txtgunicorn app:app
Node.jsnpm installnode index.js
Node.js (TS)npm install && npm run buildnode dist/index.js
Docker(auto from Dockerfile)(defined in CMD)
💡

Always bind to 0.0.0.0 and use $PORT. Render dynamically assigns the port. Hardcoding 8000 or localhost will cause deploy failures.

02

Environment Variables

Never hardcode API keys. Render lets you set environment variables per service via the dashboard or render.yaml. They're injected at build and runtime.

Adding variables in the dashboard

Service page → Environment tab
Go to your service → click Environment tab on the left sidebar.
Add Key-Value pairs
Click Add Environment Variable. Enter your key (e.g. BOT_TOKEN) and value. Click Save Changes.
Trigger a redeploy
Env var changes don't auto-redeploy. Go to Manual Deploy → Deploy latest commit.
Use Secret Files for large configs
For JSON credentials or large configs, use Secret Files (also in the Environment tab). They're mounted as files in your container.

Reading in Python

config.py
import os from dotenv import load_dotenv load_dotenv() # loads .env locally, ignored in production BOT_TOKEN = os.getenv("BOT_TOKEN") SUPABASE_URL = os.getenv("SUPABASE_URL") SUPABASE_KEY = os.getenv("SUPABASE_KEY") WEBHOOK_URL = os.getenv("RENDER_EXTERNAL_URL") # Render injects this automatically if not BOT_TOKEN: raise ValueError("BOT_TOKEN is not set!") # crash early if missing

03

The Keep-Alive Fix

Free Render services spin down after 15 minutes of no HTTP traffic. The next request triggers a cold start (~30 seconds). For bots and APIs this means dropped messages and slow responses.

🚨

This is the #1 gotcha on the free tier. Telegram webhook updates that arrive during cold start are lost — Telegram will retry a few times but if your service doesn't respond fast enough, the update is dropped.

The fix: a health endpoint + external pinger

main.py — health endpoint
from fastapi import FastAPI app = FastAPI() @app.get("/health") async def health(): return {"status": "ok"} # UptimeRobot or cron-job.org pings this every 5–10 minutes # keeping the service warm with zero cost

Option A — UptimeRobot (recommended)

1

Create account at uptimerobot.com

Free account gives you 50 monitors. No credit card needed.

2

New Monitor → HTTP(s)

URL: https://your-app.onrender.com/health. Monitoring interval: 5 minutes.

3

Enable email/Telegram alerts

Get notified when your service actually goes down (not just wakes up). Free on all plans.

Option B — cron-job.org

1

Sign up at cron-job.org

Free, unlimited cron jobs. No credit card.

2

Create job → URL: your /health endpoint

Set schedule to every 10 minutes. Method: GET. That's it.

💡

Use both. UptimeRobot for downtime alerts + cron-job.org as a backup pinger. They complement each other and both are free.

04

Webhooks & Bot Setup

Use webhook mode instead of polling for bots on Render. Polling runs a loop that wastes CPU and still can't beat webhook's simplicity. With webhooks, Telegram pushes updates directly to your endpoint.

Full webhook setup (python-telegram-bot)

main.py — webhook server
import os from fastapi import FastAPI, Request from telegram import Update, Bot from telegram.ext import Application, CommandHandler BOT_TOKEN = os.getenv("BOT_TOKEN") WEBHOOK_URL = os.getenv("RENDER_EXTERNAL_URL") # auto-set by Render app = FastAPI() ptb = Application.builder().token(BOT_TOKEN).build() async def start(update: Update, context): await update.message.reply_text("👋 Hello! I'm alive.") ptb.add_handler(CommandHandler("start", start)) @app.on_event("startup") async def startup(): await ptb.initialize() await ptb.bot.set_webhook(f"{WEBHOOK_URL}/webhook") @app.post("/webhook") async def webhook(request: Request): data = await request.json() update = Update.de_json(data, ptb.bot) await ptb.process_update(update) return {"ok": True} @app.get("/health") async def health(): return {"status": "ok"}
⚠️

One bot = one webhook URL. You can't set the same bot token to multiple webhook URLs. If you redeploy to a new Render URL, update the webhook: https://api.telegram.org/bot<TOKEN>/setWebhook?url=<NEW_URL>/webhook

Check / delete webhook manually

browser URL bar (replace TOKEN)
# Check current webhook info https://api.telegram.org/botTOKEN/getWebhookInfo # Delete webhook (switch to polling) https://api.telegram.org/botTOKEN/deleteWebhook # Set new webhook https://api.telegram.org/botTOKEN/setWebhook?url=https://your-app.onrender.com/webhook
05

render.yaml (Infrastructure as Code)

Commit a render.yaml to your repo and Render reads it to auto-configure your services. No clicking through the dashboard — deploy everything from code.

render.yaml — full example
services: - type: web name: my-telegram-bot runtime: python plan: free buildCommand: "pip install -r requirements.txt" startCommand: "uvicorn main:app --host 0.0.0.0 --port $PORT" healthCheckPath: /health envVars: - key: BOT_TOKEN sync: false # secret — fill in dashboard - key: SUPABASE_URL sync: false - key: SUPABASE_KEY sync: false - key: PYTHON_VERSION value: "3.11.0" autoDeploy: true # redeploy on every push to main
💡

Set healthCheckPath in your render.yaml. Render will use this to determine if your deploy succeeded. Without it, Render considers a deploy successful as soon as the process starts — even if it crashes 2 seconds later.

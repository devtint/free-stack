01

The Spin-Down Problem

Free tier hosting services save costs by spinning down containers when they're not receiving traffic. Here's what happens per platform:

🖥️

Render Free

Spins down after 15 minutes of inactivity. Cold restart takes 30–60 seconds. Incoming requests during startup are queued or dropped.

🚂

Railway (hobby)

Doesn't sleep, but has a $5/month credit limit. Monitor usage to avoid unexpected cutoffs.

🐍

PythonAnywhere free

Web apps must be manually reloaded every 3 months unless you have a paid plan.

🐙

GitHub Actions (public)

Scheduled workflows are disabled after 60 days of no repo activity. Push a commit to re-enable.

🚨

For Telegram bots, spin-down is critical. If your Render service is sleeping when Telegram sends a webhook update, the update is lost forever (Telegram retries for ~5 minutes then gives up). Keep your bot warm at all times.

The universal solution: a /health endpoint

Every service you deploy should expose a /health endpoint. External pingers hit this every 5–10 minutes. That's it.

FastAPI / Flask
# FastAPI @app.get("/health") async def health(): return {"status": "ok", "service": "bot"} # Flask @app.route("/health") def health(): return {"status": "ok"} # Express (Node.js) app.get('/health', (req, res) => res.json({ status: 'ok' }))

UptimeRobot Setup

uptimerobot.com — Free account: 50 monitors, 5-minute check interval, email/Telegram/Slack alerts when your service goes down. This is the most widely-used keep-alive solution.

1

Sign up at uptimerobot.com

Free account — no credit card. Email verification required.

2

Add New Monitor → HTTP(s)

Friendly Name: My Bot - Keep Alive
URL: https://your-app.onrender.com/health
Monitoring Interval: 5 minutes

3

Set up Telegram alerts (optional but great)

My Settings → Alert Contacts → Add Alert Contact → Telegram. Follow the bot setup to link your Telegram account. Get notified instantly when your service goes down.

4

That's it — monitor is active

UptimeRobot pings your /health every 5 min. Render stays warm. You get an alert if it ever actually goes down.

💡

50 free monitors = you can keep 50 Render services alive simultaneously. More than enough for any solo builder's portfolio.

cron-job.org Setup

cron-job.org — Free, unlimited cron jobs that make HTTP requests on a schedule. More flexible than UptimeRobot — you can hit POST endpoints, send headers, and run any schedule.

1

Sign up at cron-job.org

Free, no credit card. They've been reliable for years.

2

Create Cronjob → Enter your URL

URL: https://your-app.onrender.com/health
Schedule: Every 10 minutes
Method: GET

3

Enable notifications on failure

Notification → Send email on failure. Useful as a second alert layer alongside UptimeRobot.

Advanced: ping with custom schedule

cron-job.org schedule examples
# Every 10 minutes */10 * * * * # Every 5 minutes (most aggressive, recommended for bots) */5 * * * * # Every hour at :30 (for less critical services) 30 * * * * # Weekdays 9am-6pm only (save pings for when users are active) */10 9-18 * * 1-5

GitHub Actions Approach

Use GitHub Actions scheduled workflows as a keep-alive. Free for public repos (unlimited minutes). Best for triggering jobs, not just pings.

.github/workflows/keepalive.yml
name: Keep Services Alive on: schedule: - cron: '*/10 * * * *' workflow_dispatch: # allow manual trigger jobs: ping: runs-on: ubuntu-latest steps: - name: Ping Render bot run: | response=$(curl -s -o /dev/null -w "%{http_code}" \ "${{ secrets.RENDER_APP_URL }}"/health) echo "Response: $response" if [ "$response" != "200" ]; then echo "::error::Service returned $response" exit 1 fi
⚠️

GitHub Actions cron has a minimum interval of 5 minutes, and scheduled jobs on repos with no activity for 60 days are paused automatically. Don't rely on this as your only keep-alive mechanism.

Best strategy: use all three

Supabase Free Tier Tricks

Supabase free projects pause after 7 days of inactivity. Here's how to prevent it and get the most out of the free plan.

💤

Pause after 7 days

Free projects pause if no API calls are made for 7 days. Same keep-alive solutions apply — ping /rest/v1/ every day.

📊

500 MB DB limit

Includes your tables + indexes. Use VACUUM regularly. Delete old rows. Don't store binary data in DB — use Storage instead.

🔌

2 free projects

Only 2 active projects on free plan. Pause one if you need to create another. Paused projects don't count toward the limit.

⚡

Edge Functions

500k invocations/month free. Use for webhooks, background tasks, and scheduled jobs instead of a separate server.

Ping Supabase to prevent pausing

.github/workflows/ping-supabase.yml
name: Ping Supabase (prevent pause) on: schedule: - cron: '0 12 * * *' # once a day at noon UTC jobs: ping: runs-on: ubuntu-latest steps: - name: Ping Supabase REST API run: | curl -s \ -H "apikey: ${{ secrets.SUPABASE_ANON_KEY }}" \ -H "Authorization: Bearer ${{ secrets.SUPABASE_ANON_KEY }}" \ "${{ secrets.SUPABASE_URL }}"/rest/v1/ \ > /dev/null echo "Supabase pinged!"
💡

One daily ping is enough to prevent Supabase project pausing. Don't overdo it — you have 500k edge function invocations/month to spend on actual features.

01

Project Setup & Keys

Supabase gives you a full Postgres database, auth system, file storage, and realtime — all from a single project. Free tier includes 500 MB database, 1 GB storage, and 50k monthly active users.

Creating your project

1

Go to supabase.com

Sign in with GitHub. Click New Project. Choose an organization (your personal org works fine).

2

Name + region + password

Pick a project name, select the region closest to your users, and set a database password. Save it — you'll need it for direct DB connections.

3

Wait ~2 minutes

Supabase provisions your Postgres instance. You'll see a dashboard once it's ready.

4

Get your API keys

Go to Settings → API. You need three things: Project URL, anon public key, and service_role key.

Your keys — know the difference

Key Use For Safe in browser?
anon Frontend client, respects RLS policies YES
service_role Backend/server only — bypasses all RLS NEVER
🚨

Never expose service_role in frontend code or public repos. It bypasses all Row Level Security and gives full database access. Use it only in server-side code (Render, Edge Functions).

Install the client

terminal
# JavaScript / TypeScript npm install @supabase/supabase-js # Python pip install supabase
lib/supabase.ts
import { createClient } from '@supabase/supabase-js' const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL! const supabaseKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY! export const supabase = createClient(supabaseUrl, supabaseKey)

Environment variables

VariableValue SourceWhere
SUPABASE_URLSettings → API → Project URLFrontend + Backend
SUPABASE_ANON_KEYSettings → API → anon keyFrontend (safe)
SUPABASE_SERVICE_KEYSettings → API → service_roleBackend only
02

Database & Row Level Security

Supabase runs PostgreSQL 15. You get a full SQL editor in the dashboard, auto-generated REST and GraphQL APIs, and the most important feature: Row Level Security (RLS).

⚠️

Always enable RLS on every table. Without it, anyone with your anon key can read/write all data. Enable it first, then add policies.

Create a table with SQL Editor

SQL Editor — create table
-- Create a posts table CREATE TABLE posts ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE, title text NOT NULL, body text, created_at timestamptz DEFAULT now() ); -- Enable RLS ALTER TABLE posts ENABLE ROW LEVEL SECURITY; -- Policy: users can only read their own posts CREATE POLICY "Users see own posts" ON posts FOR SELECT USING (auth.uid() = user_id); -- Policy: users can insert their own posts CREATE POLICY "Users create own posts" ON posts FOR INSERT WITH CHECK (auth.uid() = user_id);

Querying with the JS client

queries.ts
// SELECT with filter const { data, error } = await supabase .from('posts') .select('id, title, created_at') .eq('user_id', userId) .order('created_at', { ascending: false }) .limit(20) // INSERT const { data, error } = await supabase .from('posts') .insert({ title: 'Hello', body: 'World', user_id: userId }) .select() .single() // UPDATE const { error } = await supabase .from('posts') .update({ title: 'Updated' }) .eq('id', postId) // DELETE const { error } = await supabase .from('posts') .delete() .eq('id', postId)

Python client (for backend / bots)

db.py
import os from supabase import create_client supabase = create_client( os.getenv("SUPABASE_URL"), os.getenv("SUPABASE_SERVICE_KEY") # service_role for server-side ) # Insert a record response = supabase.table("users").insert({ "telegram_id": 123456789, "username": "devtint" }).execute() # Query response = supabase.table("users").select("*").eq("telegram_id", 123456789).execute() user = response.data[0] if response.data else None
03

Authentication & Sessions

Supabase Auth handles email/password, magic link, OAuth (Google, GitHub, etc), and phone OTP — all free for up to 50k monthly active users.

Auth methods at a glance

📧

Email + Password

Classic signup/login. Enable email confirmation in Auth settings. Supabase sends the email via built-in SMTP.

✨

Magic Link

Passwordless. User enters email → gets a one-time login link. Zero friction, great UX for simple apps.

🔗

OAuth (Social)

Google, GitHub, Discord, Twitter. Set up OAuth app credentials and add them in Auth → Providers.

📱

Phone OTP

SMS verification via Twilio or Vonage (requires external account). Good for apps needing phone-based identity.

Sign up & login (JS)

auth.ts
// Sign up with email + password const { data, error } = await supabase.auth.signUp({ email: 'user@example.com', password: 'super-secret', }) // Sign in const { data, error } = await supabase.auth.signInWithPassword({ email: 'user@example.com', password: 'super-secret', }) // OAuth (Google) const { error } = await supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: `${window.location.origin}/auth/callback` } }) // Get current user const { data: { user } } = await supabase.auth.getUser() // Sign out await supabase.auth.signOut() // Listen to auth state changes supabase.auth.onAuthStateChange((event, session) => { if (event === 'SIGNED_IN') console.log('Welcome!', session?.user) if (event === 'SIGNED_OUT') console.log('Goodbye!') })
💡

Next.js tip: Use @supabase/ssr package with cookie-based sessions for Server Components. The client-side only SDK uses localStorage which doesn't work with SSR.

04

File Storage

Supabase Storage gives you 1 GB free with S3-compatible buckets. Files are served via CDN. You can make buckets public (no auth needed to read) or private (RLS enforced).

Setup

1

Create a bucket

Go to Storage → New bucket. Name it (e.g. avatars). Toggle Public for open assets (images, PDFs). Keep it private for user-confidential files.

2

Set storage policies

In private buckets, add RLS policies so users can only access their own files. Use the policy editor in the dashboard or SQL.

3

Upload & retrieve

Use the client SDK to upload. Public files have a permanent CDN URL. Private files use signed URLs with expiry.

storage.ts
// Upload a file const { data, error } = await supabase.storage .from('avatars') .upload(`public/${userId}.png`, file, { contentType: 'image/png', upsert: true }) // Get public URL (public buckets) const { data } = supabase.storage .from('avatars') .getPublicUrl(`public/${userId}.png`) // → data.publicUrl // Signed URL (private buckets) — expires in 60s const { data, error } = await supabase.storage .from('receipts') .createSignedUrl(`${userId}/file.pdf`, 60) // Delete a file const { error } = await supabase.storage .from('avatars') .remove([`public/${userId}.png`])

Realtime Subscriptions

Supabase Realtime broadcasts database changes (INSERT, UPDATE, DELETE) over WebSocket. Build live chats, dashboards, and collaborative apps without polling.

realtime.ts
// Listen to all INSERT events on 'messages' table const channel = supabase .channel('messages-feed') .on( 'postgres_changes', { event: 'INSERT', schema: 'public', table: 'messages' }, (payload) => { console.log('New message:', payload.new) setMessages(prev => [...prev, payload.new]) } ) .subscribe() // Always unsubscribe when component unmounts return () => supabase.removeChannel(channel)
💡

Enable Realtime on your table: In the dashboard go to Database → Replication → supabase_realtime and toggle your table on. Without this step, you won't receive any events.

Broadcast (presence / cursor sharing)

broadcast.ts
// Room-based broadcast (no DB required) const channel = supabase.channel('room:lobby') channel .on('broadcast', { event: 'cursor' }, ({ payload }) => { updateCursor(payload.userId, payload.x, payload.y) }) .subscribe() // Send your cursor position channel.send({ type: 'broadcast', event: 'cursor', payload: { userId, x: e.clientX, y: e.clientY } })

Keep-Alive & Free Tier Gotchas

The free Supabase tier has one critical limitation: projects pause after 1 week of inactivity. A paused project returns errors until manually resumed in the dashboard — or auto-woken by an API call.

⚠️

Free tier limits — 500 MB database, 1 GB storage, 50k MAU, 2 GB bandwidth. Projects pause after 7 days of no API activity. Upgrade to Pro ($25/mo) for always-on.

Keep-alive with cron-job.org (free)

1 · Create a lightweight health endpoint
In your backend (Render), add a GET /health route that makes a simple Supabase query (e.g., count rows in a tiny table).
2 · Register on cron-job.org
Free account, unlimited cron jobs. Create a new job pointing to your health endpoint.
3 · Set schedule to every 5 days
Run it before the 7-day window closes. Cron expression: 0 9 */5 * * (every 5 days at 9am).
health.py — FastAPI endpoint
from fastapi import APIRouter from db import supabase router = APIRouter() @router.get("/health") async def health_check(): # Ping Supabase to prevent project from pausing supabase.table("_health").select("count").execute() return { "status": "ok" }

Repo Setup & Best Practices

A clean repo structure saves you pain later. Start right or refactor forever.

Essential files in every repo

FilePurpose
.gitignoreNever commit node_modules/, .env, __pycache__/, .venv/, dist/
README.mdWhat is this? How do I run it? What does it do? Write it for someone who just cloned it.
LICENSEAdd MIT if open source. GitHub has a picker. Without it, technically no one can use your code.
.env.exampleTemplate of required env vars with placeholder values. Commit this. Never commit .env.
requirements.txt / package.jsonPinned dependencies. Use pip freeze > requirements.txt or npm install --save.

Good .gitignore for Python + Node projects

.gitignore
# Python .env .venv/ __pycache__/ *.pyc *.pyo *.egg-info/ dist/ # Node node_modules/ .next/ .cache/ *.log # OS .DS_Store Thumbs.db # Secrets *.pem *.key client_secret*.json
🚨

If you accidentally commit a secret, changing it in a new commit is NOT enough — git history is public. You must: 1) invalidate/rotate the key immediately, 2) use git filter-branch or BFG Repo Cleaner to rewrite history, 3) force push. Or just delete and recreate the repo.

02

GitHub Actions CI/CD

GitHub Actions runs workflows on every push, PR, or schedule. Free: 2,000 min/month on private repos, unlimited on public repos.

Auto-deploy to Render on push

.github/workflows/deploy.yml
name: Deploy to Render on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - name: Trigger Render Deploy run: | curl -X POST "${{ secrets.RENDER_DEPLOY_HOOK }}" # Render deploy hook URL: Dashboard → Service → Settings → Deploy Hooks

Run tests on every PR

.github/workflows/test.yml
name: Tests on: pull_request: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.11' - run: pip install -r requirements.txt - run: pytest --tb=short env: BOT_TOKEN: ${{ secrets.BOT_TOKEN }} SUPABASE_URL: ${{ secrets.SUPABASE_URL }}

Scheduled cron job (keep-alive alternative)

.github/workflows/keepalive.yml
name: Keep Render Alive on: schedule: - cron: '*/10 * * * *' # every 10 minutes jobs: ping: runs-on: ubuntu-latest steps: - name: Ping health endpoint run: curl -f "${{ secrets.RENDER_APP_URL }}"/health
⚠️

GitHub Actions scheduled jobs on public repos can be paused if the repo has no activity for 60 days. Use UptimeRobot or cron-job.org as a more reliable keep-alive for production bots.

03

Secrets Management

GitHub Secrets are encrypted env vars available to Actions workflows. They're never exposed in logs and can't be read once set.

Adding secrets

1

Repo → Settings → Secrets and variables → Actions

Click New repository secret. Name: BOT_TOKEN, Value: your actual token.

2

Reference in workflow with ${{ secrets.NAME }}

GitHub replaces it at runtime. It shows as *** in logs.

3

For org-wide secrets, use Organization Secrets

Settings → Secrets at the org level. Available to all repos in your org.

💡

Use vars.* for non-secret config (like APP_NAME or RENDER_URL). GitHub has a separate Variables tab — these are visible in logs. Use secrets.* only for actual sensitive values.

04

GitHub Pages

Host static HTML/CSS/JS sites for free. Custom domain + HTTPS included. Perfect for portfolios, docs, and landing pages.

Deploy a static site

1

Repo → Settings → Pages

Source: Deploy from a branch. Branch: main, folder: / (root) or /docs.

2

Your site is live at username.github.io/repo-name

Takes ~60 seconds to propagate after first setup.

3

(Optional) Add custom domain

Add a CNAME file in your repo containing your domain (e.g. myapp.com). Then add a CNAME DNS record at your registrar pointing to username.github.io.

Deploy via Actions (for built sites like React)

.github/workflows/pages.yml
name: Deploy to GitHub Pages on: push: branches: [main] permissions: contents: read pages: write id-token: write jobs: deploy: runs-on: ubuntu-latest environment: name: github-pages url: ${{ steps.deployment.outputs.page_url }} steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: 20 } - run: npm ci && npm run build - uses: actions/upload-pages-artifact@v3 with: { path: dist } - id: deployment uses: actions/deploy-pages@v4

Branch Protection

Even as a solo builder, protecting your main branch prevents accidental force pushes and ensures CI passes before merging.

Recommended rules for solo projects

✅

Require status checks

Block merges to main unless your test workflow passes. No broken code on main.

🚫

Prevent force pushes

Disables git push --force on main. Protects history from accidental rewrites.

📋

Require PR for changes

Work on feature branches, open PRs. Even solo, this gives you a review step before merging.

🔒

Include admins

Apply rules to admins too — so you can't bypass your own protection rules by accident.

Setup

Repo → Settings → Branches → Add branch protection rule → Branch name pattern: main.

💡

Conventional Commits help you auto-generate changelogs. Format: feat: add webhook, fix: handle 404, chore: update deps. Tools like release-please can auto-create GitHub Releases from these commit messages.
